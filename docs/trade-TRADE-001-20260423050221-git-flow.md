# Рабочий процесс Git и версионирование

Дата: 2026-04-23

Ветка изменения: `TRADE-001`

Версия проекта после изменения: `0.1.0`

## Назначение

Этот документ фиксирует новый обязательный порядок работы с торговым проектом.

Проект больше не правится напрямую в `master`. Любое изменение сначала выполняется в отдельной ветке, затем коммитится и только после проверки вливается обратно в `master`.

## Порядок работы

1. Перейти в проект:

```bash
cd /home/user/trade
```

2. Перед любым изменением обновить `master`:

```bash
git checkout master
git pull --ff-only origin master
```

3. Создать новую ветку:

```bash
git checkout -b TRADE-001
```

Номер ветки каждый раз увеличивается на единицу: `TRADE-001`, `TRADE-002`, `TRADE-003`.

4. Все правки выполнять только в созданной ветке.

5. После завершения проверить изменения и закоммитить.

Формат сообщения коммита:

```text
ИМЯ_ВЕТКИ - краткое описание на русском до 8 слов
```

Пример:

```text
TRADE-001 - Введены ветки и версии
```

6. После успешного коммита слить ветку в `master`:

```bash
git checkout master
git merge --no-ff TRADE-001
git push origin master
```

## Декомпозиция

Если задача большая, ее нужно делить на несколько веток.

Одна ветка должна содержать один логический блок изменений. Например:

- отдельная ветка для исправления runner'ов;
- отдельная ветка для изменения торговой логики;
- отдельная ветка для документации;
- отдельная ветка для тестов или инфраструктуры.

## Версионирование

Версия хранится в файле `VERSION`.

Формат версии:

```text
MAJOR.MINOR.PATCH
```

Правила:

- `PATCH` повышается при небольшом исправлении или багфиксе.
- `MINOR` повышается при изменении логики, процесса или крупном изменении в нескольких файлах без нарушения совместимости.
- `MAJOR` повышается при несовместимом изменении или необходимости ручной миграции.

Текущее изменение вводит новый процесс работы и затрагивает несколько файлов, поэтому версия повышена до `0.1.0`.

## Журнал изменений

Все изменения фиксируются в `change.log`.

Каждая запись должна содержать:

- версию;
- дату;
- тип изменения: `PATCH`, `MINOR` или `MAJOR`;
- ветку;
- краткий список изменений.

## Документация

Если изменение крупное, нужно добавлять отдельный документ в `docs`.

Формат имени:

```text
[проект]-[ветка]-[таймстамп]-[описание до 3 слов].md
```

Пример:

```text
trade-TRADE-001-20260423050221-git-flow.md
```

Вся проектная документация пишется на русском языке.
