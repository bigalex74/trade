# TRADE-009: Стабилизация Gemini runtime для трейдеров

Дата: 2026-04-23

Версия: `0.4.2`

## Инцидент

Пользователь заметил, что при доступности старших Gemini-моделей трейдеры иногда используют младшие модели и получают timeout.

## Что показали логи

За последние волны было видно:

- prompt'ы трейдеров компактные: примерно 2.2-3.9K символов;
- RAG формировался корректно;
- старшие модели реально работают: `gemini-3.1-pro-preview` успешно отвечала за 14-28 секунд;
- одновременно две первые задачи волны иногда получали timeout на `gemini-3.1-pro-preview` и `gemini-3-flash-preview`;
- после первого timeout health guard сразу ставил модель в cooldown для всей категории `trader`;
- следующие трейдеры уже не могли пробовать старшие модели и деградировали к младшим.

## Причина

Проблема была не в доступности модели как таковой.

Цепочка была такая:

```text
2 параллельных Gemini-вызова -> timeout на старших моделях -> мгновенный общий cooldown модели -> следующие трейдеры пропускают старшие модели -> fallback до младших моделей
```

Это слишком агрессивно для live-трейдинга: одиночный timeout не должен сразу запрещать модель всей волне.

## Исправление

1. Снижена параллельность Gemini-вызовов для трейдеров:

```bash
GEMINI_CONCURRENCY=1
```

Изменено в:

- `run_ai_job_dispatcher.sh`;
- `run_ai_trader_once.sh`.

2. Отключен мгновенный cooldown после одиночного timeout:

```bash
AI_MODEL_COOLDOWN_TIMEOUT_MINUTES_TRADER=0
```

Повторные проблемы остаются защищены через:

```bash
AI_MODEL_FAILURE_THRESHOLD=3
AI_MODEL_HEALTH_WINDOW_MINUTES=30
```

То есть одна зависшая попытка больше не отравляет модель для всей волны, но модель с повторяющимися сбоями все равно будет временно пропускаться.

## Почему не убраны младшие fallback-модели

Младшие модели оставлены как аварийный fallback.

Но нормальный путь теперь должен чаще доходить до старших моделей, потому что:

- нет параллельного давления на Gemini CLI;
- нет мгновенного общего cooldown после одного timeout;
- `ollama/llama3.2` уже отключен в `TRADE-008`.

## Проверки

Пройдены:

```bash
./tests/run_trader_runtime_config_smoke.sh
./tests/run_ai_fallback_smoke.sh
```

## Ожидаемый эффект

- меньше каскадной деградации на младшие модели;
- меньше ложных сообщений о недоступности моделей;
- старшие модели не будут пропускаться после одного случайного timeout;
- цена изменения: AI-вызовы трейдеров станут более последовательными, зато стабильнее.

## Следующий контроль

После ближайшей торговой волны нужно проверить:

```sql
SELECT trader_name, model_id, status, duration_seconds, created_at
FROM trading.ai_call_log
WHERE category = 'trader'
ORDER BY created_at DESC
LIMIT 50;
```

Критерий нормальной работы: большинство успешных решений должны приходить от старших моделей, а `model_skipped` по старшим моделям должен перестать появляться пачками после одиночного timeout.
