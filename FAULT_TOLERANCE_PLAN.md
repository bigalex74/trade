# ПЛАН ПОВЫШЕНИЯ ОТКАЗОУСТОЙЧИВОСТИ (FAULT TOLERANCE)

Поскольку локальные LLM (Ollama) исключены из-за нехватки ресурсов, система полностью зависит от внешних API (Gemini) и стабильности внутренних узлов. 

## 1. Управление процессами (Process Management)
**Текущая проблема:** Скрипты запускаются через `bash` и `nohup`. При падении скрипта (например, из-за таймаута сети) он умирает навсегда.
**Решение:**
- **Systemd / Docker Compose:** Перевод всех воркеров под управление менеджеров процессов с политикой `Restart=always`.
- **Active Watchdog:** Текущий `ai_watchdog.py` только "лает" (шлет алерты). Нужно дать ему права "кусать" — автоматически перезапускать зависшие сервисы.

## 2. Сетевая отказоустойчивость (API & LLM)
**Текущая проблема:** Если Gemini недоступен, система пытается дергать локальные модели, которых нет, а затем падает.
**Решение:**
- **Circuit Breaker (Предохранитель):** Если API Gemini возвращает ошибку 5xx три раза подряд, система делает паузу на 5 минут, чтобы не спамить запросами.
- **Exponential Backoff:** При сетевой ошибке повторять запрос через 1, 2, 4, 8 секунд.
- **Multi-Cloud Fallback:** Добавить резервные ключи от других облачных провайдеров (Groq, OpenAI) как последний эшелон защиты.

## 3. Отказоустойчивость данных (Data Resilience)
**Текущая проблема:** Если биржа (MOEX или Binance) отдала ошибку, пайплайн падает, и трейдеры остаются без данных.
**Решение:**
- **Stale Data Cache (Кэш устаревших данных):** Если свежие данные недоступны, использовать последний успешный снимок рынка с флагом `is_stale=True`.
- **Trading Halt:** Если данные устарели более чем на 15 минут, роботы автоматически переводятся в режим "Только удержание" (запрет на открытие новых позиций).

## 4. Устойчивость Базы Данных (Database Layer)
**Текущая проблема:** Синхронные блокировки при записи от 11+ агентов.
**Решение:**
- **Safe Transactions:** Обернуть все операции записи в строгие блоки `try...except` с автоматическим `ROLLBACK` при ошибке, чтобы предотвратить Deadlocks.

## 5. Изоляция сбоев (Bulkhead Pattern)
**Решение:** Разнести ядро матчинга, пайплайны данных и агентов. Зависание агента при обращении к ИИ не должно останавливать работу ядра, которое проверяет стоп-лоссы.
