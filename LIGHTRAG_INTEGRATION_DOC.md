# Документация архитектуры проекта: Knowledge-Driven Translation (V2)

## 1. Обзор системы
Проект представляет собой автоматизированный конвейер перевода новелл, усиленный графом знаний (Knowledge Graph RAG). Система не просто переводит текст, а ведет "базу знаний" мира новеллы, что исключает путаницу в именах, поле персонажей и лоре.

## 2. Компоненты системы

### А. n8n (Оркестратор)
*   **Роль:** Управление логикой, чтение файлов, разбивка на части, вызов LLM.
*   **Новая функция:** Синхронизация данных между БД и Графом знаний.

### Б. PostgreSQL (Линейная память)
*   **Таблицы:** `document_jobs`, `document_chunks`, `document_log`.
*   **Зачем:** Хранение истории перевода, статусов задач и исходных текстов.

### В. LightRAG (Семантическая память)
*   **Адрес:** `https://lightrag.bigalexn8n.ru`
*   **Технология:** GraphRAG (извлечение сущностей и связей).
*   **Зачем:** Хранение контекста мира. Позволяет находить информацию не по дате, а по смыслу.

### Г. Ollama (Локальный движок)
*   **Адрес:** `https://ollama.bigalexn8n.ru`
*   **Зачем:** Бесплатная генерация эмбеддингов (векторов) для LightRAG и вспомогательные задачи (форматирование JSON).

---

## 3. Схема взаимодействия (Data Flow)

1.  **Триггер:** Пользователь загружает файл через n8n.
2.  **Preprocessing:** Текст разбивается на чанки (500-800 токенов).
3.  **Context Enrichment (NEW):**
    *   n8n отправляет запрос в LightRAG: *"Найди информацию о персонажах в этом тексте"*.
    *   LightRAG возвращает: *"Ли Синь (мастер меча), Город Ветра (разрушен)"*.
4.  **Translation:** n8n отправляет в LLM (DeepSeek через Polza.ai):
    *   Системный промпт + **Контекст из LightRAG** + Предыдущий чанк + Текущий текст.
5.  **Post-processing (NEW):**
    *   Переведенный текст записывается в Postgres.
    *   **Этот же текст** отправляется в LightRAG для обновления графа знаний (автоматическое извлечение новых фактов).

## 4. Настройка API для n8n

### Интеграция с LightRAG
Для работы из n8n используйте узел **HTTP Request**:
*   **Method:** `POST`
*   **URL:** `https://lightrag.bigalexn8n.ru/query` (для поиска) или `/documents/upload` (для обучения).
*   **Authentication:** Basic Auth (bigalex / qQ08102003).
*   **Body (Query):**
    ```json
    {
      "query": "Кто такой основной антагонист и какие его цели?",
      "mode": "hybrid"
    }
    ```

## 5. План безопасного внедрения

### Этап 1: Подготовка (Текущий статус)
*   [x] Развертывание LightRAG и Ollama.
*   [x] Настройка SSL и внешнего доступа.
*   [x] Оптимизация скорости (40 сек/глава).

### Этап 2: Создание "Пассивного" конвейера
Мы добавим в n8n блоки записи в LightRAG, но **не будем** использовать их результаты для перевода.
*   **Цель:** Наполнить базу знаний первыми 10-20 главами и проверить качество графа.

### Этап 3: Создание "Активного" контекста
Когда граф наполнен, мы меняем воркфлоу `sub_get_context`, чтобы он подмешивал данные из LightRAG в промпт перевода.

## 6. Глобальные переменные (Рекомендация)
Рекомендуется создать в n8n следующие переменные (через раздел Variables или Environment):
1. `LIGHTRAG_URL` = `https://lightrag.bigalexn8n.ru`
2. `POLZA_API_KEY` = `pza_...`
3. `OLLAMA_URL` = `https://ollama.bigalexn8n.ru`

Это позволит обновлять ключи в одном месте без риска поломать воркфлоу.
