# ПЛАН ВНЕДРЕНИЯ ГИБРИДНОЙ RAG-СИСТЕМЫ

Статус на 2026-04-23: базовая реализация выполнена в ветке `TRADE-004`, затем prompt-контракт ужат в ветке `TRADE-006`, версия `0.3.0`.

Реализовано:

- Qdrant-память сделок в коллекции `trade_setups_memory_v1`.
- Qdrant-память новостей в коллекции `market_news_memory_v1`.
- Локальные embeddings через Ollama `nomic-embed-text` по умолчанию.
- Hash embeddings для тестов без зависимости от Ollama.
- Индексатор `ai_memory_indexer.py`.
- Wrapper `run_ai_memory_indexer.sh`.
- Регулярный cron-refresh Qdrant каждые 15 минут в торговые часы.
- Интеграция компактного `HYBRID RAG CONTEXT` в `ai_paper_trader.py`.
- Smoke-тест `tests/run_hybrid_rag_smoke.sh`.

Принцип реализации: Qdrant не принимает торговые решения. Он возвращает только короткую память похожих исходов и свежий новостной контекст. Финальное решение остается за трейдером, risk engine и matching engine.

## 1. Текущая ситуация
Анализ кода показал, что `lightrag-algo` сейчас используется как концептуальная инструкция в промптах, но не имеет глубокой интеграции с операционными данными (новости, тики).

## 2. Обоснование Standard RAG
Для биржевой торговли критична скорость и объем "сырых" фактов. 
- **Standard RAG (Vector DB):** Идеален для новостного потока и поиска похожих графиков.
- **LightRAG (Graph DB):** Идеален для хранения торговых правил и взаимосвязей между активами.

## 3. Предлагаемый стек
- **Qdrant:** Векторная база данных (высокая производительность).
- **FastEmbed:** Локальная генерация эмбеддингов (0 затрат на токены).
- **LiteLLM:** Унифицированный интерфейс для вызова Gemini/Ollama.

## 4. Схема работы
1. **Пайплайн новостей** -> Векторизация -> **Qdrant**.
2. **Трейдер** -> Запрос в Qdrant ("Что пишут про BTC?") -> Получение топ-5 новостей.
3. **Трейдер** -> Запрос в LightRAG ("Как торговать VSA?") -> Получение стратегии.
4. **Итоговый промпт:** Новости + Стратегия + Рыночные данные -> Gemini.

## 5. Ожидаемый эффект
- Сокращение контекста (не нужно слать всё, только релевантное).
- Возможность "обучения на лету": система запоминает профитные сетапы в Vector DB.

## 6. Guardrails реализации

- Индексируются не только прибыльные, но и убыточные закрытые сделки, чтобы не создавать survivorship bias.
- Каждая точка Qdrant имеет стабильный UUID от `source_key`.
- Повторная индексация идемпотентна через `source_hash`, который учитывает provider/model embeddings.
- При refresh коллекции удаляются устаревшие точки, которые больше не попадают в текущее окно lookback.
- Cron-refresh идет через `run_ai_memory_indexer.sh` и `flock`.
- Новости ограничиваются TTL через `AI_RAG_NEWS_MAX_AGE_HOURS`.
- Размер блока в prompt ограничивается `AI_RAG_MAX_CHARS`.
- Штатный лимит RAG-контекста: `420` символов.
- Штатный target всего prompt трейдера: `5200` символов при hard-limit `8000`.
- Трейдеры получают компактный `MKT`, а risk engine продолжает получать полный `market_features`.
- Если prompt все равно выходит за лимит, RAG-блок автоматически ужимается или отключается.
- При ошибке Qdrant/Ollama трейдер продолжает работу без RAG-блока.
- Операционная документация пишется в общую KB, не в ALGO KB.
